# Analyzing Test Results

5 tests ran. 1 passed. 4 failed.

Before drawing conclusions, it's worth understanding exactly what Keploy
generated and why the failures are more informative than they first appear.

---

## Why the Tests Failed

There are two distinct failure modes here, and they are worth separating clearly.

### 1. Mock mismatch (Redis)

During replay, Keploy substitutes real Redis calls with recorded mocks.
Mocks are matched by request signature. If the outgoing Redis call
differs slightly from what was recorded (different key, different OTP value)
the mock does not match, and the dependency call fails entirely.

This is what caused the `500 / EOF` responses on the OTP generation tests.

### 2. Dynamic values in responses

OTPs and JWT tokens are generated fresh on every request. Even when the app
runs correctly, the response will never exactly match the recorded one because:

- OTP: randomly generated each time
- JWT token: includes timestamps and session data

Keploy compares responses field by field. A different OTP or token results in a failed test.

<Callout type="info">
  This is not a bug in Keploy. It is a fundamental challenge with
  **deterministic replay in non-deterministic systems**. Keploy provides
  configuration options like field-level noise filtering to handle this
  but that is beyond the scope of this tutorial.
</Callout>

---

## Why test-set-1 / test-2 Passed

This was the "wrong OTP" test case, where the POST request sent an incorrect
OTP and the expected response was:

```json
{"status":"false","message":"OTP didn't match, email isn't verified"}
```

This response is **fully deterministic** — no OTP, no token, no dynamic data.
The same input always produces the same output. Keploy replayed it perfectly.

This confirms Keploy works exactly as designed. The tool is not broken.
The system under test has dynamic behavior that needs additional handling.

---

## Key Takeaways

**What worked well:**
- Keploy captured real API traffic without any test code
- Mocks for Redis were generated automatically
- Deterministic responses were replayed and verified correctly
- The entire test suite ran in under 11 seconds

**What needs attention:**
- OTP and token fields need noise filtering configuration
- Stateful Redis mocks require careful session management
- Dynamic systems need explicit handling for non-deterministic fields

---

## Final Thought

The 1 passing test is more meaningful than it looks. It proves the pipeline
works end to end: request capture, mock generation, replay, and comparison
all functioned correctly. The 4 failures are not test infrastructure problems.
They are signals about where the application produces non-deterministic output.

That distinction, between a broken test tool and a dynamic system, is
exactly the kind of understanding that makes API testing effective in practice.
